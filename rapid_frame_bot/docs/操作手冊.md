## 1. 系統概觀

這是一個全自動的單進程 asyncio 交易機器人，在 Bitunix Futures 執行「做多急速框」策略。

**運作流程一句話：** 訂閱 K 線 → 偵測型態 → 掛限價進場單 → 成交後自動掛 TP/SL → 分批出場 → 回到等待。

**兩種運行模式：**

| 模式 | `testnet` 設定 | 行為 |
|------|---------------|------|
| 紙上交易 | `true` | 所有下單只寫 Log，不送真實 API；以 K 線價格模擬成交，可完整驗證策略邏輯 |
| 實盤 | `false` | 真實下單、真實資金 |

> **建議先用紙上交易模式跑一段時間**，確認訊號與出場邏輯符合預期，再轉實盤。

## 2. 事前準備

1. **Python 3.11 以上**（開發機實測亦相容 3.14）。
2. **Bitunix 帳戶與 API 金鑰**（API Key + Secret Key）。
   - 實盤需開啟合約交易權限。
   - 建議 API 金鑰僅授予「交易」權限，**不要**開啟「提現」權限。
3. **一個 SMTP 寄信帳號**（用於通知）。若用 Gmail，需開啟兩步驟驗證並產生「應用程式密碼」，填入設定檔的 `password`。
4. （部署用）**Docker 與 Docker Compose**，若以容器方式運行。

## 4. 設定檔詳解

所有可調參數集中在 `config.yaml`。**此檔已列入 `.gitignore`，不會進版控**（避免金鑰外洩）。

```yaml
api:
  key: "YOUR_API_KEY"          # Bitunix API Key
  secret: "YOUR_SECRET_KEY"    # Bitunix Secret Key
  testnet: true                # true = 紙上交易（只寫 Log）；false = 實盤

trading:
  risk_per_trade: 0.02         # 每單最大風險 = 帳戶可用資金 × 此比例（0.02 = 2%）
  margin_mode: "cross"         # 保證金模式（全倉）
  max_concurrent_positions: 7  # 全域同時最多持倉幣種數

symbols:
  major: ["BTCUSDT", "ETHUSDT"]      # 主流幣清單
  major_leverage: 200                # 主流幣槓桿
  major_timeframes: ["1h", "3h", "1d"]  # 主流幣監控週期
  altcoin_leverage: 20               # 山寨幣槓桿
  altcoin_timeframes: ["3h", "3d"]   # 山寨幣監控週期
  min_volume_usdt: 5000000           # 山寨幣篩選門檻（24h 成交量）

email:
  smtp_host: "smtp.gmail.com"  # SMTP 伺服器
  smtp_port: 587               # SMTP 連接埠（587 = STARTTLS）
  sender: "bot@example.com"    # 寄件信箱
  password: "YOUR_APP_PASSWORD"  # 寄件信箱密碼（Gmail 用應用程式密碼）
  recipient: "you@example.com" # 收件信箱

logging:
  level: "INFO"                # 日誌等級：DEBUG / INFO / WARNING
  file: "logs/bot.log"         # 日誌檔路徑
```

**關鍵參數說明：**

- `risk_per_trade`：硬性風控。每筆交易的虧損（進場價到止損價的距離 × 數量）不超過可用資金的這個比例。改大 = 倉位變大、風險變高。
- `max_concurrent_positions`：場上同時最多開幾種幣的倉。達上限後新訊號會被忽略並記 Log。
- `*_leverage`：槓桿越高，強平價越貼近進場價。本機器人用**全倉（cross）**降低高槓桿下的強平風險，但每單 2% 風控仍是硬性上限。
- `testnet`：**最重要的開關。** 改成 `false` 前請完成第 11 節的檢查清單。

## 5. 啟動與運行

### 5.1 直接執行

```bash
cd rapid_frame_bot
# 編輯 config.yaml 填入金鑰與設定（首次）
python main.py
```

啟動後機器人會：
1. 讀取帳戶可用資金（紙上模式仍會嘗試讀取）。
2. 建立各（幣種 × 週期）狀態機。
3. 設定槓桿與全倉模式（紙上模式僅寫 Log）。
4. 恢復現有持倉（若有）。
5. 訂閱 WebSocket K 線與帳戶頻道，開始運作。

### 5.2 確認運作中

觀察日誌應看到連線、訂閱、以及每根 K 棒收盤的處理紀錄。紙上模式下，下單動作會以 `[PAPER]` 前綴出現。

## 6. 運作原理（操作視角）

> 🔒 **策略內容（專有，未公開）**
>
> 進場型態、Fibonacci 價位比例、分批出場觸發與比例、止損移位規則等策略核心，
> 屬本專案的專有內容，已從公開版本移除。

機器人對每個（幣種 × 週期）維護一台狀態機，其**架構性**狀態流轉如下（策略判定以中性描述代替）：

```
IDLE  ──🔒 策略型態偵測（專有）──►  PATTERN_FOUND
                                        │
PATTERN_FOUND ──🔒 策略失效判定──► IDLE（取消進場單）
                                        │ 進場單成交
                                        ▼
                                   IN_POSITION
                                    ├─ 分批 TP 成交 → 依策略管理止損／分批出場
                                    └─ 止損觸發 → 取消 TP → IDLE
```

**多週期規則（架構）：** 同一幣種跨所有監控週期最多 1 筆倉位；任一週期先觸發者開倉，
其餘忽略，直到平倉。監控週期由設定檔決定。

**3h K 線（架構）：** Bitunix 無原生 3h，由 1h K 棒即時合成。

## 7. 監控：日誌與 Email

### 7.1 日誌

日誌寫入 `config.yaml` 指定的 `logging.file`（預設 `logs/bot.log`）。

```bash
# 即時觀察
tail -f logs/bot.log
```

調整 `logging.level` 為 `DEBUG` 可看到更詳細的訊息（含未識別訂單、型態取消等）。

### 7.2 Email 通知（7 種事件）

| 事件 | 何時發送 |
|------|---------|
| 進場成交 | 限價進場單成交 |
| 分批止盈成交 | 各分批 TP 成交 |
| 剩餘部位平倉 | 剩餘部位依策略條件市價全平 |
| 止損觸發 | 止損成交 |
| API 連線錯誤 | 重試失敗後 |
| 當日無訊號 | 每日 UTC 23:55，若當日無任何符合型態 |
| 帳戶資金不足 | 可用資金刷新為 0 時 |

> Email 發送失敗只會記 Log、不會中斷交易。請先寄一封測試確認 SMTP 設定正確（尤其 Gmail 的應用程式密碼）。

## 8. Docker 部署

```bash
cd rapid_frame_bot

# 1. 填好 config.yaml

# 2. 啟動（背景常駐，斷線自動重啟）
docker compose up -d

# 3. 觀察即時日誌
docker compose logs -f

# 4. 停止
docker compose down
```

`docker-compose.yml` 已將 `config.yaml` 以唯讀掛載、`logs/` 以可寫掛載，時區設為 UTC。容器設定 `restart: unless-stopped`，異常退出會自動重啟。

## 9. 日常維運

- **重啟安全：** 機器人啟動時會查詢現有持倉並還原為 `IN_POSITION` 狀態，繼續管理 TP/SL，不會把既有倉位變孤兒。
  - **限制：** 若重啟前已有部分 TP 成交，純靠交易所查詢無法精確還原 tp1/2/3 的命中狀態（待日後本地狀態持久化強化）。重啟後請人工確認持倉與掛單一致。
- **資金刷新：** 每 300 秒自動刷新帳戶可用資金供倉位計算使用；資金為 0 會發信告警並暫停開倉。
- **修改設定：** 編輯 `config.yaml` 後需重啟機器人才會生效。
- **WebSocket 斷線：** 自動指數退避重連（1s→2s→…→最長 60s）並重新訂閱，無需人工介入。

## 10. 疑難排解

| 症狀 | 可能原因與處理 |
|------|---------------|
| 啟動即報認證錯誤 | API Key/Secret 錯誤，或系統時間不準（簽名含毫秒時戳）。校正主機時間。 |
| 收不到 Email | SMTP 主機/埠/帳密錯誤；Gmail 需用「應用程式密碼」。查 Log 中的 `Email failed`。 |
| 一直「WS 斷線」重連 | 網路問題或訂閱過多。確認對外連線；Bitunix WS 速率上限 5 訊息/秒。 |
| 偵測到型態但不開倉 | 已達 `max_concurrent_positions`、該幣已有倉、或可用資金為 0（倉位算出為 0）。查 Log。 |
| 紙上模式卡住不動 | 確認 `testnet: true` 且有收到 K 線；紙上成交以 K 棒 OHLC 觸發，需價格實際觸及進場價。 |
| 帳戶資金顯示為 0 | 帳戶確實無可用資金，或 API 回傳欄位名稱與預期不符（實盤前須核對，見第 11 節）。 |

## 11. 轉實盤前檢查清單

把 `testnet` 改成 `false` 前，**務必逐項確認**：

- [ ] 已用紙上模式跑過足夠時間，訊號與出場行為符合預期。
- [ ] **核對 Bitunix 實際 API 回傳欄位名稱**——本機器人對下列欄位採防禦性假設，須以實盤回應驗證：
  - 認證簽名格式（query/body 串接規則、Header 名稱）。
  - 帳戶資金欄位（`data.available`）。
  - 持倉欄位（`positionId`、`avgOpenPrice`、`qty`）。
  - 掛單欄位（`orderId`、`orderType`、`side`、`reduceOnly`、`price`）。
  - 下單回應的訂單 ID 路徑。
- [ ] 確認 WebSocket 公開／私有頻道 URL 與訂單欄位對應最新官方文件。
- [ ] 以小額資金、低槓桿先試單，確認真實下單、成交回報、止損移位皆正常。
- [ ] Email 通知實測可收到。
- [ ] 確認 `risk_per_trade` 與槓桿符合你的風險承受度。
- [ ]（建議）為「部分 TP 已成交後重啟」加上本地狀態持久化，再投入較大資金。

## 12. 安全與風險提醒

- **金鑰保護：** `config.yaml` 與 `API_KEY.txt` 已列入 `.gitignore`。切勿把含金鑰的檔案推上版控或分享。API 金鑰建議僅開交易權限、關閉提現。
- **高槓桿風險：** 高槓桿設定下，雖採全倉降低強平風險，但市場劇烈波動仍可能造成重大虧損。倉位以固定風險比例控管，但連續虧損會累積。槓桿與風險比例由設定檔決定。
- **限價進場特性：** 進場為限價單（GTC）。若市場快速突破而未成交，機器人不追價。
- **殘餘倉位：** 部分分批出場後的殘餘倉位，策略僅在特定條件下才平出，期間可能長時間持有，建議人工留意。（🔒 具體條件為策略內容，未公開。）
- **自動化責任：** 這是自動交易程式，請在你能承受的資金範圍內使用，並定期人工檢視持倉與帳戶狀態。

## 13. 多交易所設定

本機器人現可同時運行多家交易所，跑同一套「急速框做多」策略，各家獨立倉位與本金告警。

- **設定檔：** 複製 `config.example.yaml` 為 `config.yaml`，在 `exchanges:` 區塊列出各家設定。
  每家以 `enabled: true/false` 決定是否啟用——只跑 Bitunix 就把其他家設 `false`。
- **金鑰：** 不寫在 `config.yaml`。依 `secrets/README.md`，**環境變數優先**
  （`{交易所大寫}_API_KEY` / `_API_SECRET`，如 `BITUNIX_API_KEY`），缺則回退讀
  `secrets/<交易所名>.txt`。
- **目前支援：** Bitunix（手刻 REST/WS）、BingX/Binance/Bybit/OKX（透過 ccxt）。
  ⚠️ BingX 等 ccxt adapter 的止損參數與 K 線收盤判斷尚待 `scripts/probe_ccxt.py`
  實連校正，實盤前務必先以 `testnet: true` 紙上模擬驗證。
- **架構：** `Coordinator`（main.py）並行運行多個 `ExchangeSession`，單一家異常會被隔離
  並發信告警，不影響其他家。
