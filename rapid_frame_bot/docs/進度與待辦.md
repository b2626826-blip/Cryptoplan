## 真實 API 發現（REST，2026-06-23 實連唯讀 `get_account` 所得）

技術文件這段（pp. 認證簽名）整體正確，但**端點表與 query 串接規則有 3 處錯誤**，已修 `api/rest.py`：

1. **GET query 串接：** 簽名用的 query 是 ASCII key 升序後串接 **`key+value`（無 `=`、無 `&`）**，
   例 `{"id":1,"uid":200}` → `"id1uid200"`。原程式用 `key=value` 是錯的
   （`get_account` 帶 `marginCoin` 必踩；`get_klines`/`get_tickers`/`get_pending_orders` 同樣受影響）。
2. **`get_account` 端點：** 真實路徑是 **`GET /api/v1/futures/account`** 且**必帶 `marginCoin`**（如 `USDT`）。
   技術文件寫的 `/account/get_single_account` 不存在。
3. **回應格式：** `{"code":0,"msg":"Success","data":{...}}`，`data` 是**單一物件（dict）**，
   餘額欄位為 `data.available`（字串）。`main.py` 的 `acc["data"]["available"]` 讀法正確、測試亦相符。
4. **簽名/Header 確認正確：** `SHA256(nonce+timestamp+apiKey+query+body)` → `SHA256(digest+secret)`；
   Header 為 `api-key` / `nonce` / `timestamp` / `sign`（+ `Content-Type: application/json`）。
5. **`get_pending_positions`**（路徑 `/api/v1/futures/position/get_pending_positions` ✅ 與程式一致）：
   `data` 是**裸 list**（無持倉時 `[]`）；`symbol`/`positionId` 為選填 query。
6. **`get_pending_orders`**（路徑 `/api/v1/futures/trade/get_pending_orders` ✅ 與程式一致）：
   `data` 是 **dict** `{"total":"0","orderList":[]}`，掛單在 **`data.orderList`**，**非裸 list**。
   → 已修 `main.py` 消費端（原本把 `data` 當 list 迭代會壞）；測試亦改用真實形態。

驗證腳本：`python scripts/probe_rest.py`（唯讀，打上述三端點，從 `API_KEY.txt` 取金鑰，不下單）。
實測：三端點皆 `code=0`；`available≈144.62 USDT`、持倉 0、掛單 0。

## 真實 API 發現（私有 WS，2026-06-23 實連 Bitunix 所得）

1. **私有 WS 網址：** `wss://fapi.bitunix.com/private/`（與先前推定一致，已驗證可連）。
2. **登入簽名公式與 REST 不同**（已實作 `BitunixWS.sign_login`）：
   `digest = SHA256(nonce + timestamp + apiKey)`（**第一層無 query、無 body**），
   `sign = SHA256(digest + secretKey)`。timestamp 為毫秒**字串**。
3. **連線流程：** 連上後伺服器先送 `{"op":"connect","data":{"result":true}}`（同公開 WS）；
   收到後送 `{"op":"login","args":[{apiKey,timestamp,nonce,sign}]}`，**緊接著**送訂閱
   `{"op":"subscribe","args":["order"]}`（args 是頻道名**字串** list，與公開 WS 的
   `[{symbol,ch}]` 不同）。
4. **伺服器不回 login／subscribe 回執**：對照實驗（正常簽名 vs 全 0 錯誤簽名）兩者反應
   **完全相同**——皆無回應、皆不斷線。故無法被動判斷認證成敗，**只能靠真實訂單事件推送確認**。
   程式因此不等 login 回執，收到 `connect` 即送 login + 訂閱。
5. **訂單推送格式**（已修 `parse_order_message`）：頂層 `ch="order"`、`ts`、`data`；
   `data` 內狀態欄位是 **`orderStatus`**（非 `status`！值 INIT/NEW/PART_FILLED/CANCELED/
   FILLED/PART_FILLED_CANCELED），另有 `event`（CREATE/UPDATE/CLOSE）、`orderId` 等。
6. **心跳：** 伺服器會送 `{"op":"ping","ping":<ts>}`，需回 `{"op":"pong","pong":<ts>}`（已處理）；
   websockets 函式庫亦自動回應協定層 ping。8 秒煙霧測試連線穩定。

驗證腳本：`python scripts/probe_private_ws.py`（唯讀；`--bad-sign` 可做對照實驗）。

## 真實 API 發現（公開 WS，2026-06-23 實連 Bitunix 所得）

1. **正確的公開行情 WS 網址：** `wss://fapi.bitunix.com/public/`（結尾斜線必要）。
   - 技術文件原填的 `wss://fstream.bitunix.com/...` 為**不存在的網域**（nslookup NXDOMAIN）。
   - REST 主機 `https://fapi.bitunix.com` 經 nslookup 確認**正確**（Cloudflare）。
2. **真實 K 線推送格式**（與技術文件假設不同，已修正 `api/ws.py`）：
   ```json
   {"ch":"market_kline_60min","symbol":"BTCUSDT","ts":<毫秒>,
    "data":{"o":..,"c":..,"h":..,"l":..,"b":<基礎量>,"q":<計價額>}}
   ```
   - 時間在**頂層 `ts`**（伺服器時間），不在 `data.t`。
   - 成交量是 `data.b`（基礎量），不是 `data.v`。
   - **沒有 `closed` 收盤旗標**。
3. **收盤偵測機制：** Bitunix 每約 2 秒推一次「當前形成中的 K 棒」完整快照；
   跨入新週期（以 `ts // interval` 分桶）時，上一根即收盤。
   已新增 `KlineCloseDetector`（`api/ws.py`）處理，桶跳號時回傳對齊週期的收盤 K 棒。
4. **連線後第一則訊息：** `{"op":"connect","data":{"result":true}}`（連線確認，正確被忽略）。
