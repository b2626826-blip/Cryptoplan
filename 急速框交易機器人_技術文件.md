```
rapid_frame_bot/

│   └── ws.py                # WebSocket 客戶端（K 線訂閱、訂單回報）
│
├── strategy/                # 🔒 策略邏輯（專有，未公開）
│
├── risk/
│   └── sizer.py             # 倉位計算框架
│
└── trader/
    └── executor.py          # 進場、掛 TP、掛 SL、分批平倉、移止損
```

> 多交易所架構（`exchanges/` 抽象層、`session.py`、`Coordinator`）見根目錄 `README.md`。

### 2.2 資料流圖

```
Bitunix WS
  │  市場 K 線
  ▼
ws.py（含收盤偵測）
  │  已收盤 K 棒事件
  ▼
candle_builder.py ──（1h 合成 3h）──► 虛擬 K 棒
  │
  ▼ K 棒收盤事件
state_machine.py（每個 symbol × timeframe）
  │
  ├─── IDLE → 🔒 策略型態偵測（專有）
  │
  ├─── PATTERN_FOUND → 🔒 策略失效判定（專有）
  │                  → executor.py → 掛限價進場單
  │
  └─── IN_POSITION（WS 訂單頻道通知成交）
           │
           ├── 進場成交 → executor.py → 設 SL，掛分批 TP
           ├── TP 成交 → executor.py → 依策略管理止損／分批
           └── SL 成交  → 取消剩餘 TP 掛單 → 回到 IDLE
```

## 3. 模組說明

### 3.1 `api/rest.py` — REST 客戶端

**責任：** 所有需要認證的 HTTP 請求。

**主要方法：**

| 方法 | Endpoint | 說明 |
|------|---------|------|
| `get_trading_pairs()` | GET /market/trading_pairs | 取得所有合約清單 |
| `get_tickers(symbols)` | GET /market/tickers | 取得 24h 成交量（篩選用） |
| `get_klines(symbol, interval, limit)` | GET /market/kline | 取得歷史 K 線（初始化用） |
| `get_account()` | GET /account | 取得帳戶餘額（須帶 `marginCoin`） |
| `change_leverage(symbol, leverage)` | POST /account/change_leverage | 設定槓桿 |
| `change_margin_mode(symbol, mode)` | POST /account/change_margin_mode | 設定全倉/逐倉 |
| `place_order(...)` | POST /trade/place_order | 掛單（進場或 TP 分批平倉） |
| `cancel_orders(symbol, order_ids)` | POST /trade/cancel_orders | 取消掛單 |
| `place_position_tpsl(...)` | POST /tp_sl/place_position_tp_sl_order | 設定倉位止損止盈 |
| `modify_position_tpsl(...)` | POST /tp_sl/modify_position_tp_sl_order | 修改止損 |
| `get_pending_positions()` | GET /position/get_pending_positions | 查詢現有持倉 |
| `get_pending_orders(symbol)` | GET /trade/get_pending_orders | 查詢現有掛單 |

**認證簽名（雙層 SHA256）：**

```python
nonce     = uuid.uuid4().hex            # 32 位隨機字串
timestamp = str(int(time.time() * 1000))

# GET：query params 依 ASCII key 升序排列後串接（key+value，無 '='、無 '&'）
# POST：body JSON 去除所有空白後串接

digest = SHA256(nonce + timestamp + api_key + query_params + body)
sign   = SHA256(digest + secret_key)
```

**Request Headers：**
```
api-key:   <API Key>
nonce:     <random 32 chars>
timestamp: <milliseconds>
sign:      <signature>
Content-Type: application/json
```

### 3.2 `api/ws.py` — WebSocket 客戶端

**責任：** 維持長連線，分發市場與帳戶事件，並偵測 K 棒收盤。

**連線端點：**
```
公開 K 線：wss://fapi.bitunix.com/public/
私有帳戶：wss://fapi.bitunix.com/private/
```

**私有頻道（帳戶事件）：**
- `order` — 訂單成交、取消事件（狀態欄位為 `orderStatus`）

**收盤偵測：** Bitunix 每約 2 秒推一次「當前形成中的 K 棒」完整快照，且**不帶收盤旗標**；
`KlineCloseDetector` 以 `ts // interval` 分桶，跨入新週期時回傳上一根對齊週期的收盤 K 棒。

**重連機制：** 斷線後指數退避重連（1s → 2s → 4s → 最長 60s），重連後重新訂閱全部頻道。

### 3.4 `risk/sizer.py` — 倉位計算框架

以「固定風險比例」為基礎的倉位計算：由帳戶可用資金、風險比例、進場價與止損價推得數量，
並向下取整至交易所最小數量精度。

> 🔒 具體風險比例與價位來源屬策略內容，未公開。

### 3.5 `trader/executor.py` — 執行器

**責任：** 所有實際下單操作，接收狀態機指令後呼叫交易所介面（實盤）或寫 Log（紙上）。

**主要操作：**

| 操作 | 說明 |
|------|------|
| `setup_symbol(symbol, leverage)` | 設定槓桿 + 全倉模式（僅首次） |
| `place_entry(symbol, qty, entry_price, client_id)` | 掛限價進場單 |
| `cancel_entry(symbol, order_id)` | 型態失效時取消進場單 |
| `on_entry_filled(...)` | 進場成交後設 SL + 掛分批 TP |
| `move_sl_to_entry(...)` | 依策略移動止損 |
| `cancel_tp_orders(symbol, order_ids)` | SL 觸發後清除剩餘 TP 掛單 |
| `close_remaining(...)` | 市價全平剩餘部位 |

> 各方法依 `config.testnet` 決定真送 API 或只寫 Log（紙上模擬）。

## 4. 資料模型

### 4.1 Candle（K 棒）

```python
@dataclass
class Candle:
    open_time: int    # 開盤時間（毫秒）
    open:  float
    high:  float
    low:   float
    close: float
    volume: float
    closed: bool = True
```

### 4.2 FiboLevels（價位集合）

```python
@dataclass
class FiboLevels:
    fib_low:  float
    fib_high: float
    sl:       float   # 止損
    entry:    float   # 進場
    tp1:      float   # 分批止盈 1
    tp2:      float   # 分批止盈 2
    tp3:      float   # 分批止盈 3
```

> 🔒 各價位由專有的比例公式計算，比例值未公開。

### 4.3 PatternContext / PositionContext（上下文）

```python
@dataclass
class PatternContext:
    c1: Candle; c2: Candle; c3: Candle; c4: Candle
    fibo: FiboLevels
    entry_order_id: str | None = None
    entry_client_id: str | None = None

@dataclass
class PositionContext:
    symbol: str
    timeframe: str
    position_id: str
    entry_price: float
    total_qty: float
    tpsl_id: str | None          # 止損訂單 ID（用於修改）
    tp_order_ids: list[str]      # 分批 TP 訂單 ID
    tp1_hit: bool = False
    tp2_hit: bool = False
    tp3_hit: bool = False
    fibo: FiboLevels = None
```

## 5. 狀態機

### 5.1 狀態定義（架構）

```
IDLE            等待型態出現
PATTERN_FOUND   已掛進場限價單，監控失效條件
IN_POSITION     持倉中，管理分批出場與止損
```

### 5.2 狀態轉移（架構，策略判定以中性描述代替）

```
┌──────────────────────────────────────────────────────────┐
│                          IDLE                            │
│  每根 K 棒收盤後：🔒 策略型態偵測                          │
│    若成立 → 計算價位 → 掛限價進場單                        │
└────────────────────┬─────────────────────────────────────┘
                     │ 型態成立 + 進場單已掛
                     ▼
┌──────────────────────────────────────────────────────────┐
│                     PATTERN_FOUND                        │
│  每根 K 棒收盤後：🔒 策略失效判定                          │
│    若失效 → cancel_entry() → 回到 IDLE                    │
└────────────────────┬─────────────────────────────────────┘
                     │ 進場限價單成交（WS 訂單頻道通知）
                     ▼
┌──────────────────────────────────────────────────────────┐
│                     IN_POSITION                          │
│  WS 訂單頻道事件：依策略管理分批出場與止損移位             │
│    SL 成交 → cancel_tp_orders() → 回到 IDLE              │
└──────────────────────────────────────────────────────────┘
```

### 5.3 全域持倉鎖（架構）

```python
# 跨週期共享，確保每幣種只有一筆倉位
active_positions: dict[str, bool]  # {symbol: True/False}

# 進入 IN_POSITION 時：active_positions[symbol] = True
# 回到 IDLE 時：       active_positions[symbol] = False
# 開倉前：若 active_positions.get(symbol) 為真 → 跳過
```

## 6. Bitunix API 對接

### 6.1 基礎資訊

| 項目 | 值 |
|------|---|
| REST Base URL | `https://fapi.bitunix.com` |
| WS Public URL | `wss://fapi.bitunix.com/public/` |
| WS Private URL | `wss://fapi.bitunix.com/private/` |
| WS 速率上限 | 5 條訊息 / 秒 |
| REST K 線上限 | 10 req/sec/ip |

> ⚠️ 官方文件曾出現數處與實際不符（WS 網域、REST query 串接規則、`get_account` 路徑、
> `get_pending_orders` 結構、訂單狀態欄位名）；上表為實連驗證後的正確值。

### 6.2 關鍵 REST Endpoints

**取得 K 線歷史（初始化用）**
```
GET /api/v1/futures/market/kline?symbol=BTCUSDT&interval=1h&limit=50
```

**掛限價單（進場 / TP 分批平倉）**
```
POST /api/v1/futures/trade/place_order

進場單 body：
{
  "symbol": "BTCUSDT", "side": "BUY", "qty": "0.2", "price": "<進場價>",
  "tradeSide": "OPEN", "orderType": "LIMIT", "reduceOnly": false,
  "effect": "GTC", "clientId": "entry_BTCUSDT_1h_<ts>"
}

TP 分批平倉 body（reduce-only）：
{
  "symbol": "BTCUSDT", "side": "SELL", "qty": "<分批數量>", "price": "<止盈價>",
  "tradeSide": "CLOSE", "orderType": "LIMIT", "reduceOnly": true,
  "effect": "GTC", "clientId": "tp1_BTCUSDT_1h_<ts>"
}
```

**設定倉位止損**
```
POST /api/v1/futures/tp_sl/place_position_tp_sl_order
{
  "symbol": "BTCUSDT", "positionId": "123456", "slPrice": "<止損價>",
  "slStopType": "MARK", "slOrderType": "MARKET"
}
```

**修改止損**
```
POST /api/v1/futures/tp_sl/modify_position_tp_sl_order
{
  "symbol": "BTCUSDT", "tpslId": "789", "slPrice": "<新止損價>",
  "slStopType": "MARK", "slOrderType": "MARKET"
}
```

### 6.3 WebSocket K 線訊息格式（實連驗證）

**訂閱請求：**
```json
{ "op": "subscribe", "args": [ { "symbol": "BTCUSDT", "ch": "market_kline_60min" } ] }
```

**推送格式（每約 2 秒一次，形成中快照）：**
```json
{
  "ch": "market_kline_60min", "symbol": "BTCUSDT", "ts": 1718000000000,
  "data": { "o": "97000", "h": "97500", "l": "96800", "c": "97400", "b": "150.5", "q": "..." }
}
```

> 時間在頂層 `ts`（非 `data.t`）；成交量為 `data.b`（基礎量）；**無 `closed` 收盤旗標**，
> 由 `KlineCloseDetector` 以分桶跳號判定收盤。

**私有 WS 訂單推送：** 頂層 `ch="order"`、`ts`、`data`；狀態欄位為 **`orderStatus`**
（值域 INIT/NEW/PART_FILLED/CANCELED/FILLED/PART_FILLED_CANCELED），另有 `event`、`orderId`。

**私有 WS 登入簽名**（與 REST 不同）：
```
digest = SHA256(nonce + timestamp + apiKey)   # 第一層無 query、無 body
sign   = SHA256(digest + secretKey)
```
> 伺服器不回 login／subscribe 回執，認證成敗只能靠真實訂單事件推送確認。

## 7. 下單流程

### 7.1 啟動初始化

```
1. get_trading_pairs() 取得所有合約
2. get_tickers() 過濾成交量不足的幣種
3. 對每個幣種：change_margin_mode(CROSS) + change_leverage()
4. get_klines() 取得歷史 K 線，初始化各狀態機的 K 棒緩衝
5. 訂閱所有幣種的 WS K 線頻道
6. 訂閱私有帳戶頻道（order）
```

### 7.2 進場 → 成交 → 出場（架構）

```
K 棒收盤 → state_machine.on_candle_close(candle)

[IDLE] 🔒 策略型態偵測 → 計算價位 → sizer 算數量 → executor.place_entry() → PATTERN_FOUND
[PATTERN_FOUND] 🔒 策略失效判定 → executor.cancel_entry() → IDLE

WS 訂單頻道：進場單 FILLED
  → executor.on_entry_filled()：place_position_tpsl(SL) + 掛分批 TP → IN_POSITION

WS 訂單頻道：TP / SL 成交
  → 依策略移動止損、記錄、或 cancel_tp_orders() → IDLE
```

> 🔒 分批出場的觸發價位與比例、止損移位規則屬策略內容，未公開。

## 9. 重要限制與注意事項

### 9.1 API 限制

| 項目 | 說明 |
|------|------|
| Bitunix 無原生 3h K 線 | 必須從 1h 自行合成 |
| WS 速率上限 5 msg/sec | 多幣種大量訂閱需批次處理 |
| REST K 線 10 req/sec | 初始化大量幣種時需加入 sleep |
| 官方文件多處失準 | 端點/格式須以真實回應核對（見第 6 節） |

### 9.2 限價進場特性

進場為限價單（GTC）。若市場快速突破而未成交，機器人不追價（等待或取消）。

### 9.3 型態重疊處理（架構）

同一幣種在不同週期同時出現型態時，依「第一個觸發者」開倉，其餘週期訊號自動忽略，
直到該倉位完全平倉後才允許新訊號。

### 9.4 開機重連處理

```
Bot 重啟時：
  1. get_pending_positions() 確認現有持倉
  2. 對有持倉的幣種直接進入 IN_POSITION 狀態
  3. get_pending_orders() 恢復 TP 訂單 ID 列表
  4. 重新訂閱所有 WS 頻道
```
> 限制：若重啟前已有部分 TP 成交，純靠交易所查詢無法精確還原命中狀態（待本地狀態持久化強化）。
